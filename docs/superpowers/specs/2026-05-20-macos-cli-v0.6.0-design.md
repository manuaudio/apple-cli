# macos-cli v0.6.0 Design Spec

**Date:** 2026-05-20  
**Status:** Approved by Manu Jimenez  
**Repo:** `~/Developer/macos-cli`

---

## Overview

v0.6.0 is a major expansion of the macos-cli project in two directions:

1. **CLI expansion** ŌĆö six new command modules + window additions + a permission system, making the binary a complete agentic macOS control surface.
2. **Integration layer** ŌĆö three optional components (MCP server, HTTP bridge, companion app) that let any LLM ŌĆö Claude, Ollama, GPT, or any OpenAI-compatible model ŌĆö drive the same CLI. All live in the same repo; users install what they need.

```
macos-cli/

```

### Installation model

- `macos` binary ŌĆö required baseline, works standalone for agents and power users
- `macos-mcp` ŌĆö optional, add to Claude Desktop's MCP config
- `macos-bridge` ŌĆö optional, run as a local daemon; point Ollama/LM Studio at it
- `macos-agent` ŌĆö optional, a menu bar app for non-technical users

All components share the same `tools.json` definition file so capabilities never drift out of sync.

## Part 1 ŌĆö CLI Expansion

### 1.1 New command modules

#### `macos spaces`
Manage macOS Mission Control Spaces via private CGS APIs (no external tools).

| Subcommand | Description |
|---|---|
| `list` | List all spaces with index and active flag |
| `switch <index>` | Switch to space by index |
| `move-window-to <index> [--app <name>]` | Move focused (or named) window to a space |
| `create` | Create a new space |
| `destroy <index>` | Destroy a space (must be empty) |

Implementation: CGSMainConnectionID + CGSCopySpaces private framework. No shelling out to yabai.

#### `macos audio`
Audio device routing, separate from the existing `volume` command.

| Subcommand | Description |
|---|---|
| `list` | List all input and output devices with IDs |
| `set-output <device-name-or-id>` | Set default audio output |
| `set-input <device-name-or-id>` | Set default audio input |
| `info` | Show current default input/output |

Implementation: CoreAudio `AudioHardware` API.

#### `macos script`
Run arbitrary JXA or AppleScript and return the result.

| Subcommand | Description |
|---|---|
| `run --jxa <code>` | Execute JavaScript for Automation and return result |
| `run --applescript <code>` | Execute AppleScript and return result |
| `run --file <path>` | Execute a .js or .scpt file |

Implementation: shells to `/usr/bin/osascript`. Timeout: 30s default, `--timeout` override.  
Use case: agent escape hatch when a purpose-built subcommand doesn't exist yet.

#### `macos timemachine`
Time Machine control without opening System Settings.

| Subcommand | Description |
|---|---|
| `status` | Show last backup time, destination, and whether a backup is running |
| `start` | Trigger an immediate backup |
| `stop` | Cancel the running backup |
| `list-snapshots` | List local snapshots with dates |
| `list-destinations` | List configured backup destinations |

Implementation: `tmutil` CLI wrapper + `com.apple.TimeMachine.plist` parsing.

#### `macos menu`
Interact with menu bar items (system tray icons, application menus).

| Subcommand | Description |
|---|---|
| `list [--app <name>]` | List menu bar items for the app (or focused app) |
| `click <menu-path>` | Click a menu item by path, e.g. `"View > Show Sidebar"` |
| `list-extra` | List extra menu bar items (status icons) |

Implementation: Accessibility API (`AXMenuBar`, `AXMenuBarItem`). Requires Accessibility TCC permission.

### 1.2 Window additions

Additions to the existing `WindowCommand`:

| Subcommand | Description |
|---|---|
| `fullscreen [--app <name>]` | Toggle fullscreen for focused or named app |
| `maximize [--app <name>]` | Zoom/maximize window to fill screen (non-fullscreen) |
| `snap --position <half\|third\|corner> [--app <name>]` | Snap window to screen region |

`snap` positions: `left`, `right`, `top`, `bottom`, `top-left`, `top-right`, `bottom-left`, `bottom-right`, `left-third`, `center-third`, `right-third`.

### 1.3 Permission System

**Design: onboarding whitelist, agent-mediated.**

On first run, `macos auth setup` is invoked by the agent (or user). It grants all capabilities to the current operator. The user can then ask the agent to revoke specific ones conversationally ("don't let the agent delete calendar events").

#### Config file

`~/.config/macos-cli/auth.json`

```json
{
  "version": 1,
  "capabilities": {
    "calendar.read": true,
    "calendar.write": true,
    "calendar.delete": false,
    "mail.read": true,
    "mail.send": false,
    "mail.delete": false,
    "contacts.read": true,
    "contacts.write": false,
    "keychain.get": false,
    "keychain.set": false,
    "reminders.read": true,
    "reminders.write": true,
    "reminders.delete": false,
    "notes.read": true,
    "notes.write": false,
    "notes.delete": false,
    "screen.capture": true,
    "screen.lock": true,
    "spaces.read": true,
    "spaces.write": false,
    "audio.read": true,
    "audio.write": false,
    "script.run": false,
    "menu.read": true,
    "menu.click": false,
    "timemachine.read": true,
    "timemachine.write": false,
    "system.shutdown": false,
    "system.reboot": false
  }
}
```

**Default policy:** reads are `true`, writes/deletes are `false`. Safe out of the box.

#### Auth.swift ŌĆö shared enforcement

```swift
struct Auth {
    static func check(_ capability: String) throws {
        // load ~/.config/macos-cli/auth.json
        // if capability is false or missing ŌåÆ throw ValidationError
    }
}
```

Every subcommand that performs a write or destructive action calls `Auth.check("capability.id")` at the top of `run()`. Read-only subcommands skip the check (reads are default-allowed).

#### `macos auth` subcommand

| Subcommand | Description |
|---|---|
| `setup` | Interactive wizard: print all capabilities with current values, prompt Y/N for each. Saves to auth.json. |
| `list` | Print all capabilities and their current allow/deny state |
| `grant <capability>` | Set a capability to `true` |
| `deny <capability>` | Set a capability to `false` |
| `reset` | Reset auth.json to default policy |

`setup` is the onboarding flow. An agent runs it on first install; a human user can run it from Terminal. The agent can also be asked conversationally ("deny mail.send") and it calls `macos auth deny mail.send`.

## Part 2 ŌĆö Integration Layer

### 2.1 Shared tool definitions (`tool-definitions/tools.json`)

A machine-readable JSON file describing every macos CLI command as a callable tool. Format: OpenAI function-calling schema (compatible with Claude tool_use, Ollama, GPT-4, etc.).

```json
[
  {
    "name": "macos_window_snap",
    "description": "Snap a window to a screen region",
    "parameters": {
      "type": "object",
      "properties": {
        "app": { "type": "string", "description": "App name (omit for focused window)" },
        "position": { "type": "string", "enum": ["left", "right", "top", ...] }
      },
      "required": ["position"]
    }
  },
  ...
]
```

This file is the single source of truth. The MCP server and HTTP bridge both read it ŌĆö no duplication.

---

### 2.2 MCP Server (`macos-mcp`)

A Swift or TypeScript MCP server that wraps the macos binary.

- Reads `tools.json` at startup and registers each entry as an MCP tool
- On tool call: constructs the correct `macos <command> <subcommand> --flags` invocation, runs it, returns stdout as the tool result
- Claude Desktop users add to `~/.config/claude/mcp_servers.json`:
  ```json
  { "macos": { "command": "/usr/local/bin/macos-mcp" } }
  ```
- Permission system applies: if auth.json denies a capability, the MCP tool call returns an error message, not an exception

---

### 2.3 HTTP Bridge (`macos-bridge`)

A local HTTP server that exposes the same tools in OpenAI function-calling format.

- Listens on `localhost:2772` (configurable)
- `GET /v1/tools` ŌĆö returns the full tool list (OpenAI format)
- `POST /v1/tool_calls` ŌĆö accepts an array of tool calls, executes them, returns results
- Compatible with: Ollama (function calling), LM Studio, Open WebUI, any OpenAI-compatible client
- Runs as a launchd daemon or as a foreground process (`macos-bridge start`)
- Auth check: reads auth.json before executing, returns `{"error": "capability denied"}` on block

---

### 2.4 Companion App (`macos-agent`)

A SwiftUI menu bar app for non-technical users who don't want to open a terminal or Claude Desktop.

- Lives in the menu bar (no Dock icon)
- Hotkey (configurable, default: Cmd+Shift+Space) opens a floating input panel
- User types natural language request
- App calls Claude Agent SDK (subscription auth via `claude auth login`, one-time setup)
- Claude receives the tool definitions from `tools.json` and calls the appropriate macos commands
- Results reported as a brief notification or in the input panel
- First launch: runs `macos auth setup` in a terminal sheet

**Auth:** Uses Claude subscription (no API key, no per-token billing). Requires the user to have a Claude plan.

## Implementation Sequence

1. **Auth system** ŌĆö `Auth.swift`, `auth.json`, `macos auth` subcommand, guard calls in existing destructive commands
2. **New CLI commands** ŌĆö `spaces`, `audio`, `script`, `timemachine`, `menu`, window additions
3. **`tools.json` generation** ŌĆö generate from source or write by hand, validate against CLI help output
4. **MCP server** ŌĆö thin wrapper, reads `tools.json`, delegates to binary
5. **HTTP bridge** ŌĆö local REST server, same delegation pattern
6. **Companion app** ŌĆö SwiftUI, Agent SDK integration, onboarding flow

---

## What This Is Not

- No Hammerspoon, yabai, Raycast, or any external dependency
- No GUI for permission management (the CLI and agent are the interface)
- No cloud component ŌĆö everything runs locally
- No cross-platform support ŌĆö macOS only, intentionally
